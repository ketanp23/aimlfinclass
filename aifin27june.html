PARTI.

The landscape — AI, ML, and where you are standing

Before any model, a working map. The terms are used loosely in industry; in a classroom they have to mean specific things.

A NOTE Every term here is a set. AI is the largest set. Deep learning is the smallest set you'll touch this morning.

The nested hierarchy

Artificial Intelligence is the umbrella discipline — any system that performs tasks usually associated with human cognition. A rule-based fraud filter is AI. So is a chess engine. So is GPT-5.

Machine Learning is the subset that learns patterns from data rather than executing hard-coded rules. Instead of writing if income < X and debt > Y then deny, you let a model discover the threshold from history.

Deep Learning is the subset of ML built on multi-layer neural networks — what powers transcription, vision, and most of what people now call "AI" colloquially. Not today's focus, but it sits inside the same workflow.

In a bank

A rules engine refuses a loan because the applicant's DTI exceeds 0.45. A logistic regression model assigns a probability of default of 0.71 and lets a credit officer set the cut-off. Both are "AI". Only one is ML. Only one updates itself when next quarter's defaults arrive.

The three paradigms of learning

ParadigmWhat you give the modelFinance example
Supervised
Most of today
Features X with known labels y.Predict next-quarter EPS from prior fundamentals. Predict default from applicant features.
UnsupervisedFeatures X only — find structure.Cluster traders by behavior. Segment customers without pre-defined personas.
ReinforcementAn environment, actions, rewards.Execution algos that learn to minimize market impact across thousands of fills.

The workflow that doesn't change

Whatever the model — regression, random forest, deep network — the seven steps are the same. Most failures in production ML are failures in steps one through three, not step five.

  1. Frame the problem. Is the target continuous (price, return) or categorical (default/no default)? This single decision picks half your toolset.
  2. Get the data. Time-stamped, point-in-time correct, no look-ahead.
  3. Clean and prepare. Missing values, outliers, encoding, scaling. The work that no one writes case studies about.
  4. Split. Train, validation, test. In time-series finance, split by time, never randomly.
  5. Train. Fit the model on the training fold.
  6. Evaluate. On data the model has not seen. The number that goes on a slide.
  7. Deploy & monitor. Drift is real. Last year's model meets next year's market.
Common trap

Look-ahead bias. Using information that wouldn't have been available at the prediction time. A model that predicts Monday's return using Tuesday's volume is "100% accurate" and worthless.

PARTII.

Regression — predicting a number

When the answer you want is a price, a return, a yield, a duration — you are doing regression. Everything else is a variation on this theme.

SHORTHAND y is what you want to know.
X is everything you have.
β is what the model learns.

The premise

Regression assumes the value you want to predict is some function of inputs, plus noise:

y = f(X) + ε y: target · X: features · ε: irreducible noise (the part no model can recover)

Your job is to estimate f. The shape you assume for f is the model.

Linear regression — the workhorse

Assume f is a straight line (or a hyperplane, with many features):

ŷ = β₀ + β₁x₁ + β₂x₂ + … + βₚxₚ β₀: intercept · β₁…βₚ: coefficient on each feature · the model is what you get when you've picked the βs

The βs are picked to minimize the squared distance between predictions and truth on the training data. This is Ordinary Least Squares, and it has a closed-form solution:

β̂ = (XTX)−1 XTy A matrix inversion. No iteration needed. Why linear regression is fast on any dataset that fits in memory.

The five assumptions (and what to do when they break)

  1. Linearity. The relationship between each feature and the target is approximately a straight line. Check: residual-vs-fitted plot. Fix: transform the feature (log, square), or move to a non-linear model.
  2. Independence of errors. Residuals don't depend on each other. Breaks loudly in time-series. Fix: use a time-aware model, or model the residuals.
  3. Homoscedasticity. The variance of errors is constant across the range of predictions. In finance, return variance often grows with the level — this assumption is the first to go.
  4. Normality of errors. Matters mostly for confidence intervals, less for the point estimate.
  5. No multicollinearity. Features aren't redundant copies of each other. Two features measuring "size" — total assets and revenue — will fight each other for the same coefficient.
Case · Equity factor model

A multi-factor return model — Fama-French style — is a linear regression. Returns regressed on market, size, value, momentum factors. The βs are factor loadings; the residual is α. The whole edifice of factor investing is OLS with a careful feature set.

Multiple linear regression — and the dummy variable trap

Real finance models never use a single feature. A loan-amount model wants income, age, tenure, credit score, and a few categorical fields like city and employment type. OLS extends to all of these in one matrix equation — but categorical features need a careful encoding step, and one specific way of doing it badly is so common it has a name.

One-hot encoding the categorical columns

OLS only speaks numbers. A column where the values are {Mumbai, Delhi, Bangalore, Chennai} must become numbers first. The standard trick is one-hot encoding: turn one column with k values into k binary columns, one per value, where exactly one is 1 for any given row.

Reading the coefficients

After fitting the corrected model, each city dummy's coefficient has a specific meaning: the effect of being in that city, relative to the baseline category. If βmumbai = 8.2, the model says a Mumbai applicant is approved for ₹8.2 lakh more than an otherwise-identical Chennai applicant, holding income, age, and everything else fixed. The baseline city itself has no coefficient — it's already in the intercept.

ŷ = β₀ + β₁·income + β₂·age + βm·mumbai + βd·delhi + βb·bangalore No chennai term — its effect lives in β₀. A Chennai applicant with income=0 and age=0 would be predicted as β₀; every other city adds (or subtracts) its own βₓ.
Spotting the trap in the wild
  • Symptom 1 · Software refuses to fit. NumPy raises LinAlgError: Singular matrix. R's lm() silently drops one dummy and warns. scikit-learn's LinearRegression returns wild coefficients (it uses a pseudoinverse and "fits" anyway, but the coefficients have no meaning).
  • Symptom 2 · Coefficients balloon when you add a dummy. Adding a fourth city dummy makes the other three's coefficients jump by orders of magnitude. The model is splitting one effect across collinear columns.
  • Symptom 3 · Standard errors explode. Variance inflation factor (VIF) above 10 on any feature is the textbook warning.
Case · Loan underwriting at a retail bank

A loan-amount model with income, age, tenure, four cities, and three employment types means 3 + 4 + 3 = 10 columns naively — but the correct OLS only uses 3 + (4−1) + (3−1) = 8. The two dropped categories (one city, one employment type) become the baselines. Every other coefficient reads relative to them, which is exactly how a credit policy committee already thinks: "all else equal, what does it cost us to lend to a self-employed Mumbai applicant versus the salaried Chennai baseline?"

Evaluation — how good is "good"?

MetricWhat it tells youWatch out for
MSE mean squared errorAverage squared miss. Punishes big errors harder than small.Units are squared — hard to interpret in business terms.
RMSE root MSEMSE in the original units. A loan-amount RMSE of $14,200 is readable.Still sensitive to outliers.
MAE mean abs. errorAverage miss, in original units, equal weight to all errors.Doesn't penalize the catastrophic miss the way risk officers want it to.

A model that beats the mean by an R² of 0.07 is doing real work in equity prediction. A model with R² of 0.92 on next-day returns is overfitting, leaking, or both. — Empirical Asset Pricing, ad lib.

PARTIII.

Linear vs non-linear — the bias-variance bargain

The decision between a straight line and a flexible curve isn't aesthetic. It is a bet about how much signal vs noise lives in your data.

TRADE Bias = how much the model misses the truth on average.

Variance = how much the model would jitter if you re-trained it on a different sample.

The trade you can't escape

Every model picks a point on a spectrum:

  • Linear / simple models have high bias (they can't fit complex shapes) and low variance (they're stable across samples). Think: a long-only equity factor model.
  • Flexible / non-linear models have low bias (they can fit anything) and high variance (they'll memorize quirks of the training sample). Think: a deep tree fit on six months of trades.

The sweet spot is the model just flexible enough to capture the real structure, not so flexible it learns the noise. Cross-validation is the tool for finding it.

Four non-linear models you should know

1 · Polynomial regression

Linear regression on transformed features. Add x², x³, interactions — solve the same OLS. You stay in the linear-coefficient world but you can fit curves.

ŷ = β₀ + β₁x + β₂x² + β₃x³ Still linear in β. Curve in x. Useful when you suspect mild non-linearity (e.g., diminishing returns to scale).

Finance use: non-linear yield curve fitting; convexity-aware bond pricing; modeling diminishing marginal utility in retail credit.

2 · K-Nearest Neighbors (KNN)

No formula. To predict for a new observation, find its k closest neighbors in the training set and average their outcomes. The model is the training data.

Finance use: peer-comparable valuation. "Find me the five most similar mid-cap industrials by margin profile and leverage; their average P/E is the fair multiple."

3 · Decision Trees

A sequence of yes/no splits on features, each split chosen to most reduce the variance (regression) or impurity (classification) of the remaining group. Reads like a flowchart.

Finance use: credit scorecards historically; transparent risk segmentation. A single tree is interpretable. An ensemble of them — a random forest — is usually the better predictor.

4 · Neural networks

Composed of layers of weighted sums followed by non-linear activations. Universal function approximators, given enough data. Beyond today's scope but worth naming — and worth knowing they are still trained on the same loss functions you will see this morning.

Choosing

Start linear. Get the simplest plausible model running end-to-end. Only when the residuals show structure the linear model can't capture should you reach for non-linear. The reverse — starting with a deep model and simplifying — is how teams ship overfit pipelines that look great in backtests and lose money live.

PARTIV.

Logistic regression — when the answer is yes or no

Despite the name, this is the canonical classification model. Used everywhere from credit scoring to medical triage to A/B testing — because it gives you a probability, and a probability is what risk decisions actually want.

NOT Logistic regression does not predict a class. It predicts a probability of a class. The threshold is a policy decision.

The sigmoid

Take a linear combination of features and squash it into [0, 1]:

p = σ(z) = 1 / (1 + e−z) where z = β₀ + β₁x₁ + … + βₚxₚ. The same linear combination as before. The σ wrapper makes it a probability.

The β coefficients are interpreted on the log-odds scale: a unit increase in feature xj multiplies the odds of the positive class by eβⱼ. A coefficient of 0.7 on "credit utilization" means the odds of default multiply by ≈2 for each unit increase in utilization.

How it is trained

Logistic regression has no closed-form solution. It is fit by minimizing the log-loss (cross-entropy) via gradient descent:

L = −(1/n) Σ [ yᵢ · log(pᵢ) + (1−yᵢ) · log(1−pᵢ) ] Reward the model for confident, correct predictions. Punish — heavily — confident wrong predictions.

Evaluation — accuracy is rarely the right number

If 2% of loans default, a model that says "no default" for everyone is 98% accurate and entirely useless. Classification evaluation needs more than one number.

MetricAnswers the questionUse when
AccuracyWhat fraction did I get right?Classes are roughly balanced.
PrecisionOf the loans I flagged as default, how many actually defaulted?The cost of a false alarm is high (e.g., turning away a good borrower).
RecallOf the actual defaults, how many did I catch?The cost of a missed positive is high (e.g., missed fraud).
F1Harmonic mean of precision and recall.You need a single number and care about both.
AUC-ROCHow well does the model rank positives above negatives across all thresholds?You haven't committed to a threshold yet, or thresholds will vary by policy.
Case · Credit decisioning

A lender's model outputs default probabilities. Risk team sets the cut-off — say 0.30 — based on expected loss vs. portfolio growth targets. Marketing pushes for 0.45 (more loans approved). Compliance pushes for 0.20 (less risk). The model didn't change. The business decision did. Logistic regression's gift is that it surfaces this conversation explicitly.

PARTV.

Where this lands in finance

A grounded inventory — the places where the models you've just seen are doing the actual work, what the inputs look like, and what tends to go wrong.

RULE OF THUMB Continuous answer → regression family.

Yes/no answer → logistic family.

If you're not sure, you haven't framed the problem yet.

Five live use cases

Credit underwriting · logistic

Predict probability of default within 12 months. Inputs: income, debt-to-income, credit score, employment tenure, prior delinquencies. Threshold tuned to portfolio loss tolerance. Re-trained quarterly with recent vintages.

Stock return forecasting · linear / non-linear

Predict next-period excess return from factor exposures and technicals. R² is intentionally low — markets are mostly efficient. The signal that exists shows up in long-short portfolios, not in individual prediction accuracy.

Asset valuation · regression, often non-linear

Estimate fair value of an illiquid asset (private equity stake, commercial real estate, structured product) from comparables. KNN-style "find similar deals" approaches still win against complex models when comparable data is rich.

Fraud detection · logistic / tree ensembles

Score each transaction's probability of being fraudulent. Extreme class imbalance (less than 1% fraud) means recall matters far more than accuracy, and precision-recall curves matter more than ROC.

Algorithmic execution · regression for impact

Predict market-impact cost of a parent order given size, urgency, ADV, spread, volatility. Used to choose between aggressive and patient execution schedules.

Choose the simplest model that meets the task. Then justify every gram of complexity you add to it. — Lab brief, after lunch

LAB01.

Build a regression pipeline

Drag nodes from the palette into the pipeline. Choose a dataset, clean it, split, fit a model, evaluate. Compare a linear and a non-linear model on the same data — the difference is the point of this lab.

📖 Lab guide — cleaning, features, and what to combine read first ▾

Cleaning runs in order. You can stack multiple cleaning nodes — each is applied to the output of the previous. Pick the operations that match the failure modes you actually see in the data preview.

Drop missing rows Removes any row containing a null. Fast and safe but throws away data. Use when nulls are few (under 5% of rows).
Mean impute Fills nulls with the column's average. Preserves row count but injects a value that didn't exist in reality — it distorts variance and biases small samples. Use when you can't afford to lose rows and the column is roughly symmetric.
Drop duplicates Removes rows that are identical across all features. Critical for any dataset where duplicate records inflate the frequency of a segment. Use when the data came from joins or ingestion pipelines (which is to say, almost always).
Remove outliers (IQR) Drops rows where any feature sits more than 1.5×IQR beyond the first or third quartile. Stabilizes linear models that get pulled by extreme values. Caution — on stock returns, extreme moves often are the signal. Skip there.
Standardize (z-score) Recentres each feature at mean=0, std=1. Essential before KNN, distance-based models, and gradient-descent training. Use when features are on different scales (income in lakhs vs. credit score in hundreds).
Normalize (min-max) Scales each feature to [0, 1]. Same goal as z-score; preferred when distributions are non-Gaussian or bounded. Use when you want bounded inputs (e.g., neural net pre-processing).
Log-transform skew Applies log(x+1) to positive features. Compresses long tails — income, loan amounts, transaction sizes — into something a linear model can fit. Use when a histogram of the feature is right-skewed.
The canonical order
dedup→ drop / impute nulls→ outlier removal→ log-transform→ scale→ feature engineering

Scaling and log-transform are last because feature engineering should see clean, comparable inputs. Outlier removal before scaling so the scale isn't dragged by extremes.

Feature engineering is how a linear model learns non-linear shapes. You don't change the algorithm — you change the columns it sees. The trade-off is overfitting: more features need more rows.

Polynomial degree 2 For each numeric feature x, adds an x² column. A linear regression on the expanded set can fit gentle curves — diminishing returns, depreciation, U-shapes. Example · housing price often grows with size but at a decreasing rate. x² captures the curve a straight line can't.
Pairwise interactions For every pair (xᵢ, xⱼ), adds an xᵢ · xⱼ column. Captures effects that exist only when two features move together — "high DTI and low credit score" is more dangerous than either alone. Example · loan amount often depends on income × tenure jointly. Either feature alone misses it.
When to use either

Polynomial when residual plots show a curve. Try it before reaching for KNN or trees.

Interactions when domain knowledge says two features should multiply (rate × duration, price × volume, income × tenure).

Always pair with z-scaling — squared and product terms blow up in magnitude. Without scaling, the OLS matrix becomes ill-conditioned and the model is unstable.

Skip both when using KNN or trees — they discover non-linearities and interactions internally. Adding polynomial features to a tree wastes splits.

These are starter recipes, not the only right answers. Compare to your own combinations and notice how each step shifts the metrics — that observation is the lab.

Housing comps — predicting price drop duplicates → drop missing → log-transform → linear regression → evaluate
Why: housing has 4 injected duplicates and ~5 nulls. Sizes range 900–3300 sqft — log compresses the tail. Linear is the right baseline since the synthetic relationship is mostly linear in size and location.
Housing comps — same data, non-linear model drop duplicates → drop missing → standardize → KNN regressor → evaluate
Why: KNN needs z-scaling or distance is dominated by sqft (the largest-magnitude feature). Compare R² to the linear recipe above — on this dataset, linear usually wins. That's the lesson.
Loan amount — heavy outliers drop missing → log-transform → IQR outliers → linear regression → evaluate
Why: loan amounts span 3L–25L+ with a few extreme cases injected. Log-transform first, then IQR — applied to the log-scale, IQR catches genuinely anomalous loans rather than just the largest ones.
Loan amount + dummies — multi-feature with categoricals drop missing → MLR + dummy fix → evaluate
Why: this dataset has city and employment one-hot encoded into 4 + 3 dummy columns. Plain Linear regression fails on it — every row's city dummies sum to 1, which makes XTX singular and the inversion impossible. Try it first to see the error; then swap in MLR + dummy fix which drops one category from each group as the baseline. Read the log to see which baselines were chosen, and remember every other coefficient is then the effect relative to that baseline.
Stock returns — the efficient market lesson drop missing → standardize → linear regression → evaluate
Why: Do not remove outliers here — extreme returns are part of the data-generating process you're trying to learn. Expect R² near zero; this is correct and is the central lesson on equity prediction.
Pipeline tips by model

Linear regression — log-transform skewed features (income, amount); remove outliers; scaling optional.

MLR + dummy fix — for any dataset with one-hot encoded categoricals. Drops one column from each dummy group as the baseline (the alphabetically-first one, matching pandas drop_first=True). Skip scaling on the dummy columns themselves — z-scoring a 0/1 column distorts the interpretation.

Polynomial regression — must z-scale first or the higher-order terms dominate numerically.

KNN regressor — z-scale or min-max is non-negotiable. The model is distance; without scaling, units of measure determine importance.

Decision tree regressor — scaling is unnecessary (splits are scale-invariant). Outlier removal can still help by giving cleaner splits.

LAB02.

Classify with logistic regression

Predict a binary outcome — default, fraud, credit-grade downgrade — and see what changes when accuracy stops being the headline metric. Linear regression is also included as a counter-example: applying it to a 0/1 target is a useful mistake to make once, on purpose.

📖 Lab guide — cleaning, features, and what to combine read first ▾

Cleaning operations run in order — each is applied to the output of the previous. For classification, the rules are slightly different than regression: you have to be careful not to throw away rare positives (defaults, fraud, downgrades).

Drop missing rows Removes rows containing any null. Fast and safe but throws away data — including, possibly, rare positive cases. Use when nulls are random; check that you don't disproportionately lose positives.
Mean impute Fills nulls with the column average. Preserves row count but the imputed value is closer to the majority class, which can flatten signal in imbalanced data. Use when you can't afford to lose positives.
Remove outliers (IQR) Drops rows where any feature is more than 1.5×IQR beyond Q1/Q3. Often counter-productive in fraud and default modeling — outliers are frequently the positives you want to catch. Caution — verify with the dataset preview that you aren't removing the rare positive class.
Standardize (z-score) Recentres each feature at mean=0, std=1. Required before logistic regression with gradient descent and before KNN. Use when features span different scales (amount in rupees vs. ratio between 0 and 1).
Normalize (min-max) Scales each feature to [0, 1]. Same goal as z-score; preferred for bounded inputs and non-Gaussian distributions. Use when features have clear hard bounds.
Log-transform skew Applies log(x+1) to positive features. Particularly useful for transaction amounts which span several orders of magnitude. Use when a feature's range crosses orders of magnitude.
The canonical order
drop / impute nulls→ log-transform (if needed)→ scale→ feature engineering

Outlier removal is conspicuously absent. In classification with rare positives, default behavior is to keep outliers — they're the cases the model exists to flag.

Feature engineering lets logistic regression — a linear model — capture non-linear decision boundaries. Without it, logistic regression can only separate classes with a straight line in feature space.

Polynomial degree 2 Adds x² for each feature. Lets logistic regression learn that risk peaks at intermediate values rather than monotonically rising. For example, default risk may be lowest for borrowers with 5–10 years employment and rise on both ends. Example · default risk by age: high for very young, low in middle years, rising again near retirement. A line can't see this — x² can.
Pairwise interactions Adds xᵢ · xⱼ for each pair. Captures compound risk — a borrower with high DTI is risky; one with high DTI and low credit score is exponentially more so. The interaction term models that compounding. Example · fraud: transaction amount × foreign matters more than either alone. A small foreign charge is normal; a large one is suspicious.
When to use either

Polynomial when a risk factor has a U-shape (very low and very high values both indicate the positive class).

Interactions when domain knowledge says compound risk exists. Most credit scorecards include interactions implicitly.

Always pair with z-scaling — squared and product terms become numerically huge without it, and gradient descent will fail to converge.

Skip for tree-based models — trees discover splits and combinations internally; pre-computed interactions don't help and waste depth.

These are starter recipes for the classification datasets in this lab. Compare against your own combinations — the metric you choose to optimize (accuracy vs. recall vs. AUC) depends on the cost of each error type, which is a business choice, not a model choice.

Loan default — the credit underwriting baseline drop missing → standardize → logistic regression → evaluate
Why: logistic regression with z-scaled features is the canonical credit model. ~18% default rate means accuracy is misleading — focus on AUC and recall in the results. Compare to "linear reg as classifier" to see why the squashing matters.
Transaction fraud — heavy class imbalance drop missing → log-transform → standardize → logistic regression → evaluate
Why: transaction amount spans orders of magnitude — log-transform first, then z-scale. ~8% fraud rate means a model that predicts "no fraud" for everyone scores 92% accuracy and catches zero fraud. Compare accuracy to AUC.
Credit downgrade — the accuracy trap drop missing → standardize → decision tree classifier → evaluate
Why: ~6% downgrade rate. A tree can achieve ~95% accuracy by predicting "no downgrade" almost always — but its F1 will collapse, exposing that high accuracy is meaningless on imbalanced data. This recipe exists to teach the accuracy trap.
Fraud — KNN comparison drop missing → log-transform → standardize → KNN classifier → evaluate
Why: KNN needs z-scaling absolutely. Compare AUC to the logistic regression recipe — KNN often loses on imbalanced data because nearest neighbors are usually the majority class. Yet another lesson about imbalance.
Pipeline tips by model

Logistic regression — z-scale is required (gradient descent needs it). Log-transform skewed features before scaling.

Linear regression as classifier — included as a counter-example. Its scores are unbounded and uncalibrated; the AUC will be lower than logistic on the same data. That comparison is the lesson.

KNN classifier — z-scale or min-max non-negotiable. Often struggles on imbalanced data.

Decision tree classifier — scaling unnecessary. Can overfit at high depth — try depth 3–5 first.
